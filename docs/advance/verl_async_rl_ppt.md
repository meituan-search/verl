### 区块 G（最右侧高亮区）：异步强化学习的影响

**标题**：`异步强化学习的影响`

**总述**：

> 异步强化学习（Asynchronous RL）通过将经验生成与模型训练完全解耦，解决了传统同步训练中因任务复杂性差异导致的 GPU
> 资源闲置问题（长尾瓶颈）。这一范式对大模型训练、推理及系统效率产生了深远影响。

**核心影响与优势**（三个紫色高亮卡片）：

- **🚀 突破算力瓶颈，提效显著**
  在大型推理模型或智能体强化学习中，生成阶段耗时极长。异步架构允许推理节点持续生成数据，训练节点独立更新权重。系统不再需要等待批次中最慢的序列生成完毕，可实现高达数倍的端到端训练加速。

- **⚡ 原生支持细粒度并行**
  支持对样本生命周期进行独立调度，环境交互与奖励计算可重叠执行，大幅减少了硬件的「计算气泡」与空闲损耗。

- **📈 容忍滞后数据，精度不降反升**
  系统设计上允许训练模块使用滞后几步的「老策略」模型所生成的数据。研究与实践表明，这种异步容忍度不仅能完全掩盖空闲开销，甚至能维持或提升模型最终的逻辑推理与数学能力。

### 区块 B（中间左）：同步训练的核心瓶颈

**标题**：`同步训练的本质困境`

**核心逻辑**：

> 当前主流强化学习算法（PPO、GRPO、DAPO 等）的迭代逻辑一致：每轮迭代依赖最新模型生成训练样本，**等所有样本生成完成后再更新模型**。这种模式契合 RL 算法定义、保障训练稳定性，但存在无法回避的效率瓶颈——**模型更新必须等待最长的样本轨迹完成**。

**长尾问题**：

> 样本轨迹长度参差不齐时，出现长尾轨迹 → 大量硬件资源闲置 → 训练效率降低 → 周期被拉长。**关键在于：这种瓶颈无法通过单纯增加硬件资源解决**，传统模式难以实现有效 scaling，成为大规模 RL 落地的核心障碍。

> 🐌 **长尾问题广泛存在于长 CoT、Agentic RL 训练中**——长尾越严重，整体效率越低。

---

**实战案例：DAPO 32B 训练曲线**

| 指标 | 数值 | 占比 |
|------|------|------|
| 单步训练总时长 | **1700 秒** | 100% |
| Rollout 阶段耗时 | **1200 秒** | **~70%** |
| 训练阶段耗时 | ~500 秒 | ~30% |

**关键发现**：

- 训练后期样本长度差异显著：**最大长度 20k vs 平均长度 5k**
- Rollout 阶段是效率优化的**关键路径**（占总时间 70%）

---

**微观视角：Qwen-235B 长尾监控**

> 以 Qwen-235B 模型训练为例（Prometheus + Grafana 监控）：
>
> - 单步 Rollout 总时长：**30 分钟**
> - **超过一半的时间在处理长尾样本**
>
> 正是这些超长轨迹的存在，让大量时间和资源耗费在等待上。

**流程图**：

```
┌─────────┐       ┌──────────┐       ┌───────┐

```

### 区块 C（中间）：传统强化学习的三大瓶颈

**标题**：`传统强化学习`

#### 01 长尾效应，资源利用率低 🐌

> 在同步训练模式下，Rollout 阶段必须等待批次中**最长序列生成完成**才能进入训练。大量 GPU 资源在等待长尾样本时处于空闲状态。在
> DAPO 32B 等大规模训练任务中，Rollout 占总时间约 **70%**，且单纯增加 rollout 资源无法有效缩短该阶段耗时——因为瓶颈是*

* *最慢的那一个样本**。

#### 02 资源耦合，调度僵化 🔗

> Trainer 和 Rollout 共享同一 GPU 资源池（Colocated
> 共置架构），无法根据实际负载动态调整两者的资源配比。推理显存占用高峰时挤占训练空间，训练反向传播时又阻塞推理请求，两者互相争抢导致整体效率下降。

#### 03 模型迭代效率制约竞争力 ⚡

> 当前大模型发布节奏持续加快，从 Llama 到 Qwen 到 DeepSeek，迭代周期不断压缩。传统的同步训练模式下，端到端训练耗时成为关键瓶颈——
**更快的训练速度意味着更快的产品迭代和市场响应能力**。

### 区块 A（左上）：One Step Off Policy（第一步异步）

**标题**：`🔄 One Step Off Policy — 第一步`

**核心思想**：

> 生成与训练并行化——使用上一步生成的样本进行当前步训练，同时异步生成下一步所需样本。打破「等全部生成完再训练」的串行依赖。

**关键技术**：

- 关闭 co-locate 模式（`hybrid_engine=False`），Trainer 与 Rollout 使用**独立 GPU 资源池**
- 基于 NCCL 的参数同步，延迟通常 **< 300ms**，对 RLHF 训练可忽略
- 连续迭代器 `_create_continuous_iterator` 实现 **1-step 预取**

**实验结果（Qwen2.5-3B, 8×A100）**：

| Engine        | Colocate Sync | One-Step-Off | 提升       |
|---------------|---------------|--------------|----------|
| VLLM+FSDP2    | 19h18m        | 15h34m       | **+23%** |
| VLLM+Megatron | 18h21m        | 13h06m       | **+40%** |

**关键成果**：相比传统同步模式，整体训练效率显著提升 **+40% ~ +70%**

---

### 区块 B（左下）：Fully Async Training（生产级架构）

**标题**：`🚀 Fully Async Training — 完全异步`

**核心思想**：

> 将 Trainer 与 Rollout 拆分为**独立 Ray Actor**，通过 MessageQueue 缓冲队列解耦数据流。支持逐样本流式生成（
`gen_batch_size=1`），实现 0.x-step 到多步的灵活异步粒度。

**关键特性**：

| 维度   | One-Step-Off    | Fully Async                          |
|------|-----------------|--------------------------------------|
| 架构   | 单体 Trainer 内部异步 | Trainer / Rollouter **独立 Ray Actor** |
| 数据传递 | Future 直接传递     | **MessageQueue 缓冲队列**                |
| 异步粒度 | 固定 1-step off   | 支持 **0.x-step 到多步**                  |
| 流式生成 | 批量生成            | **逐样本流式**                            |

**演进意义**：

> 从 One-Step-Off 的「概念验证」走向 Fully Async 的「生产级架构」——这是 verl 异步强化学习能够支撑千卡级规模训练的关键一步。


### 区块 G（右下）：Log Prob 版本一致性机制

**标题**：`Log Prob 版本一致性机制`

**问题背景**：

> 在异步训练中，Log Probability 的计算版本必须与训练使用的模型版本保持一致，否则会导致策略梯度估计偏差。

**公式表达**：

$$
\text{logprob}(t) = \log \pi_\theta(a_t | s_t, \\{a_1, ..., a_{t-1}\\})
$$

$$
\text{ref\\_logprob}(t) = \\log \\pi_{\\text{ref}}(a_t | s_t, \\{a_1, ..., a_{t-1}\\})
$$

**一致性保证**：

- Old Log Prob：使用同步时刻点的模型参数计算
- Ref Log Prob：使用 Reference Model 参数计算
- 通过 CheckpointEngine 确保版本对齐

# 第五页：昇腾核心贡献 — TransferQueue

## 标题：`昇腾核心贡献：TransferQueue — 高性能异步流式数据管理模块`

> **论文**：[AsyncFlow: An Asynchronous Streaming RL Framework for Efficient LLM Post-Training](https://arxiv.org/abs/2507.01663)（Han et al., 2025）
> **核心定位**：TransferQueue 是具备全景数据可见性和流式调度能力的高性能数据存储与传输模块，专为后训练工作流中的高效数据流优化而设计
> **GitHub**：[Ascend/TransferQueue](https://github.com/Ascend/TransferQueue) ⭐ | 已正式集成至 VeRL ([PR#5401](https://github.com/verl-project/verl/pull/5401))

---

### 区块 A（左侧）：核心价值与特性

**副标题**：`为什么需要 TransferQueue？`

**问题背景**：

> 在 VeRL 的 Single Controller 架构下，所有 `DataProto` 对象必须经过 `RayPPOTrainer` 路由，导致整个后训练系统存在**单点瓶颈**。TransferQueue 的引入正是为了**缓解这一数据传输瓶颈**。

**四大核心特性**：

| # | 特性 | 说明 |
|---|------|------|
| 🚀 | **零拷贝传输** | 避免数据序列化开销，直接内存操作 |
| 🔌 | **多协议扩展** | 支持 TCP、RDMA 等多种传输协议 |
| 📊 | **细粒度访问** | 子样本级别（sub-sample-level）的数据管理与负载均衡 |
| ⏱️ | **生命周期管理** | 主动式资源回收，支持全景数据可见性 |

---

### 区块 B（中上）：架构总览 — Control Plane + Data Plane

**副标题**：`双平面分离架构`

```
+================================================================+
|                    TransferQueue 架构                             |
+=================================================================
|                                                                  |
|  ==================Control Plane==================   Data Plane   |
|  |  TransferQueueController                    |    =======   |
|  |                                            |    |     |   |
|  |  • 全景数据追踪（生产状态 + 消费状态）      |    |Storage   |
|  |  • 消费历史记录（per computational task）   |    |Unit ×N  |
|  |  • 可定制 Sampler（Sequential / GRPOGroupN)|    |         |
|  |  • Load-balancing（实验性）                |    |         |
|  |  • Partition 隔离                          |    |         |
|  ==================↑↓=========================    =========   |
|                                                                  |
|  Training Engine ←→ TransferQueue → Inference Engine            |
|  (Producer)        (Data Bus)      (Consumer)                     |
+==================================================================
```

**Control Plane 核心能力**：
- 追踪每个训练样本的**生产状态和消费状态**
- 不同计算任务可独立消费同一份数据字段，互不干扰
- 通过 `Sampler` 抽象类支持自定义数据检索逻辑

**Data Plane 存储后端**：

| 后端 | 类型 | 特点 |
|------|------|------|
| **SimpleStorage** | 默认 | CPU 内存存储，最小格式约束，开箱即用 |
| **Yuanrong（元荣）** | Beta | 昇腾原生数据系统，HBM/DRAM/SSD 分层存储 |
| **MooncakeStore** | Beta | KV 分层存储，GPU↔DRAM 间 RDMA 传输 |
| **RayRDT** | Alpha | Ray Direct Transport，Actor 间直传 |

### 区块 D（右侧下）：两个关键设计理念

#### ① 数据流拆分 — 生产者-消费者模型

> 基于**生产者-消费者模式**构造细粒度数据流。数据生产者将 batch 数据分拣路由到 Transfer Queue，消费者按需消费。支持任意粒度的拆分分组，有效避免任务阻塞。

```
+================+           +===============+           +================+
| 数据生产者     |  分拣路由   |  Transfer      |  按需消费  | 数据消费者     |
| (Rollout)      | ────────→ │  Queue         | ───────→  | (Trainer)      |
+================+           +===============+           +================+
```

#### ② 数控分离 — 控制面与数据面解耦

> 控制信令走 Control Plane（Controller 调度），实际数据走 Data Plane（Storage Unit 直传）。数据无需经过控制节点，提供极灵活的调度路由能力。
