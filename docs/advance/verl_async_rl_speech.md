## 第二页：为什么需要全异步（2.5 分钟）

### 📖 演讲词：

那么，**为什么我们需要全异步**？让我们深入分析这个问题。

**首先，模型迭代速度决定竞争力。** 大家可以看到这个趋势图——前沿模型的发布节奏正在急剧加快，从月级压缩到周级。任何能显著缩短训练时间的技术改进，都会直接转化为市场窗口优势。

**其次，让我们深入理解同步训练的核心瓶颈。**

当前主流的强化学习算法——无论是 PPO、GRPO 还是 DAPO——它们的迭代逻辑本质上是一致的：每一轮迭代都依赖最新模型生成训练样本，**等所有样本生成完成后再对模型进行更新**。这种模式完全契合强化学习的算法定义，也能很好地保障训练过程的稳定性。但它存在一个无法回避的显著效率瓶颈——**模型更新必须等待生成阶段中最长的样本轨迹完成**。

如果生成的样本轨迹长度参差不齐、出现长尾轨迹，就会导致大量硬件资源处于闲置状态，直接制约模型的迭代效率。**而这种瓶颈无法通过单纯增加硬件资源来解决**——传统模式很难实现训练的有效 scaling，这就是大规模强化学习落地的核心障碍之一。

给大家举一个直观的案例：这是我们使用 DAPO 算法训练 32B 模型的真实数据——单步训练总时长 **1700 秒**，其中 Rollout 阶段耗时 **1200 秒**，达到了总时间的 **70%**。这是训练效率优化的关键路径。特别是到了训练后期，样本长度差异特别显著：最长的样本轨迹达到了预设最大长度 **20k**，但平均长度只有 **5k**——表现出明显的长尾生成问题。

我们再从微观角度看一下具体的长尾现象。这张图是我们通过 Prometheus + Grafana 监控到的真实情况：以 Qwen-235B 模型为例，单步 Rollout 的总时长达到了 **30 分钟**，而其中**超过一半的时间都在处理那些长尾样本**。正是这些超长轨迹的存在，让大量时间和资源都耗费在等待上。

这个流程图清晰地展示了问题所在：Rollout 生成 → **等长尾** → Train 训练。在同步模式下，短样本完成后 GPU 就在空转——1700 秒中有 1200 秒在等长尾。

总结一下传统强化学习的**三大瓶颈**：

1. **🐌 长尾效应，资源利用率低**——Rollout 占总时间约 **70%**，瓶颈是最慢的那一个样本，加资源没用
2. **🔗 资源耦合，调度僵化**——Trainer 和 Rollout 共享 GPU（Colocated 架构），互相争抢
3. **⚡ 模型迭代效率制约竞争力**——端到端耗时直接决定产品迭代速度

**实验数据佐证了这一点。** 这里的 6 张训练曲线图来自 DAPO Qwen2.5-32B 的实际训练数据：

- 异步模式 loss 收敛同样稳定 ✅
- 异步模式准确率不降反升 ✅
- 吞吐量显著提升 ✅

**最后，这张长尾分布热力图是最有说服力的证据。** X 轴是请求时间戳，Y 轴是生成长度。大家可以清晰看到——90% 的样本在 1000 token
内完成，但剩余 10% 可能延伸到 5000+ token。**哪怕长尾样本只占 1%，所有 GPU 都要等它。** 这就是我们必须做异步的根本原因。

## 第三页：基础功能构建（3 分钟）

### 📖 演讲词：

理解了"为什么"，现在来看"怎么做"。这一页介绍我们的**两大核心异步方案**。

**🔄 第一步：One Step Off Policy**

核心思想非常直观——**生成与训练并行化**。使用上一步生成的样本进行当前步训练，同时异步生成下一步所需样本。打破"
等全部生成完再训练"的串行依赖。

关键技术点包括：

- 关闭 co-locate 模式，Trainer 与 Rollout 使用**独立 GPU 资源池**
- 基于 NCCL 的参数同步，延迟通常 **< 300ms**，对 RLHF 训练可忽略
- 连续迭代器实现 **1-step 预取**

实验结果非常有说服力：在 Qwen2.5-3B 上，VLLM+FSDP2 提升 **23%**，VLLM+Megatron 提升 **40%**。整体训练效率提升 **40%~70%**。

**🚀 第二步：Fully Async Training — 完全异步**

这是从"概念验证"走向"生产级架构"的关键一步。我们将 Trainer 与 Rollout 拆分为**独立 Ray Actor**，通过 MessageQueue
缓冲队列解耦数据流。

大家看这个对比表——One-Step-Off 是单体 Trainer 内部异步，而 Fully Async 是真正的**独立 Actor + MessageQueue + 逐样本流式生成
**，支持从 0.x-step 到多步的灵活异步粒度。

**中间的大图展示了完整的 Async Stream Pipeline。** 这是 GPU 利用率的时间线——Rollout、Load Weight、Train、Prefill/Generate
各阶段通过 Partial Rollout 机制无缝衔接，几乎消除了所有计算气泡。

**资源按需配比是另一个重要特性。** 三种部署模式——推理密集型（75% Rollout）、训练密集型（75% Train）、均衡型（各
50%）——可以根据任务特点灵活选择。两者完全解耦、独立运行，**彻底消除长尾等待瓶颈**。

**Partial Rollout 支持是关键技术创新。** 核心理念是 "Make AgentLoop agnostic to rollout interruption"
——当参数同步触发时，中断活跃请求 → 保存中间状态 → 同步参数 → 恢复生成。整个过程对上层 AgentLoop 透明。

**Log Prob 版本一致性机制**解决了异步训练中的一个微妙问题：Log Probability 的计算版本必须与训练使用的模型版本保持一致，否则会导致策略梯度估计偏差。我们通过
CheckpointEngine 确保版本对齐。

**最右侧的性能数据表是我们的核心成果。** 在 Qwen3-30B-A3B 的 2K+20K 场景下，64 卡配置：

- fully async 在**所有规模下均显著快于** colocate sync
- 资源配比 64:64 时，100 step 训练从 **13h44m 降至 3h13m**，即 **~4.3x 加速**
- 准确率不降反升：async 模式 last accuracy 达到 **0.3094**，超越 sync 模式的 0.2996

## 第四页：基础设施重构（2.5 分钟）

### 📖 演讲词：

有了基础功能之后，我们需要将架构夯实为**生产级基础设施**。这一页展示四大基础架构重构。

**左侧的总览图展示了四大引擎及其关系：**

1. **① Model Engine（训练引擎）**— 支持 FSDP、Megatron Core、VeOmni 等多种后端
2. **② Rollout Engine（推理引擎）**— 接入 vLLM、SGLang、TensorRT-LLM，支持 ToolAgent、BWAgent
3. **③ TransferQueue（传输队列）**— 基于 MessageQueue 的高性能数据传输层
4. **④ Checkpoint Engine（参数同步引擎）**— 支持 naive、nccl、hccl、nixl、mooncake 等多后端

**训练引擎重构**方面，我们构建了通用的 SFT 和 RL 训练引擎。FSDP 适合快速原型（Day 0 适配），Megatron Core 性能最高（MoE
模型），VeOmni 专注多模态（约一周适配）。

**推理引擎重构**方面，Rollout Engine 构建了高度灵活的框架——Server-Client 分层架构，支持 Partial Rollout 的 abort/resume
接口，AgentLoopManager 与 LLMServerManager 完全解耦。这意味着第三方 Agent 只需消费 LLMServerClient 即可接入。

**参数同步重构**提供了丰富的通信后端选择：NCCL/HCCL 性能最高，MOONCAKE-NV 和 Mooncake 适合跨节点高带宽场景，Transfer Engine
提供统一的抽象层。

**消息传输重构**是最深刻的变革之一。从前的 MessageQueue 是单体耦合的——数据流与控制流混杂、pickle
序列化开销大。重构后实现了解耦分层——零拷贝、多协议支持、数据类型感知序列化、主从调度解耦，变为松耦合的可插拔后端架构。

## 第五页：异腾核心贡献 — TransferQueue（2 分钟）

### 📖 演讲词：

这一页重点介绍**异腾团队的核心贡献——TransferQueue**。

**TransferQueue 作为数据总线**，协调管理各个计算任务之间的数据依赖，在大规模 MoE 模型后训练中发挥了关键作用。

**左侧列出四大核心特性：**

- 🚀 **零拷贝传输**——避免数据序列化开销
- 🔌 **方便扩展**——支持多传输层、TCP、RDMA
- 📊 **细粒度数据访问**——读取/写入/追加子队列
- ⏱️ **主动式生命周期管理**

**中间的架构图展示了 TransferQueue 的核心设计——Control Plane 与 Data Plane 分离：**

- Control Plane 负责 Controller 调度和 Storage Unit 管理
- Data Plane 负责 train data 和 infer data 的高效传输
- Training Engine 和 Inference Engine 通过 TransferQueue 松耦合连接

这一设计带来了 **40%+ 的训练性能提升**。

**右侧上方介绍数据流拆分机制**——基于生产-消费模型的细粒度数据流。数据生产者将 batch 数据分拣路由到 Transfer
Queue，数据消费者按需消费。**支持任意粒度的数据流拆分分组，最大可避免任务阻塞。**

**右侧下方介绍数控分离架构**——提供独立的调度控制链路。所有待调度数据经过控制节点调度，但**实际数据无需经过控制节点**
，控制信令与数据面分离。这种设计提供了极其灵活的调度路由能力。

**底部强调：TransferQueue 已独立开源，并原生支持 verl。** 这是一个重要的生态贡献。

## 第六页：生态扩建（2.5 分钟）

### 📖 演讲词：

有了扎实的基础设施，我们开始**扩建生态边界**。

**左侧展示软硬件多维度适配能力：**

- **推理框架**：vLLM → SGLang → TensorRT-LLM，覆盖主流推理引擎
- **训练框架**：FSDP → Megatron-LM → MindSpore，覆盖 NVIDIA 和昇腾生态
- **硬件生态**：NVIDIA → Ascend → AMD → 国产芯片（寒武纪、沐曦、摩尔），实现全栈适配

**右上方的 Agent Gateway 是开放式的 Agent 接入层。** 核心问题是：当 verl 只支持固定的单轮和多轮 Agent rollout 时，新 agent
接入成本较高，难以复用。

解决方案是将 vLLM/SGLang 的推理服务抽象为**标准 OpenAI Chat Completions API**。任何兼容该协议的 Agent
系统——NeMo-Gym、Bedrock、SWE-agent、NaMoGym 等——均可无缝接入 verl 训练循环，无需修改底层代码。架构上包含 Canonical
Tokenizer、Request Router、Token-level Trajectory Recording 等核心组件。

**中间的 Elastic Scheduling 弹性调度系统**解决的是资源比不合适导致训练效率低下的问题。核心组件包括：

- **Coordinator（决策大脑）**——监控队列深度，决策切换
- **ElasticTrainer（DP Rebuild）**——动态重建数据并行组
- **ElasticActorWorker（引擎切换）**——同一 GPU 上 Train ↔ Rollout 互斥运行
- **ElasticCheckpointManager（双路同步）**——确保切换时参数一致性

持续监控生产/消费者速率，自动触发弹性节点的模式切换，快速进行资源调度。

**右下方的 Online Policy Distillation（在线策略蒸馏）**解决两步 OPD 训练中学生模型因长尾样本导致过拟合的问题。

核心思路是在 Fully Async 架构中集成 Teacher Model Manager，联合优化 RL Loss 与蒸馏 Loss：
`actor_loss = rl_loss + α × distill_loss`。Teacher Model 提供 Soft Labels，Student Model 进行 RL Training，通过 Joint
Optimization 和 Weight Refitting 实现联合优化。

关键优势是：**异步 OPD 训练中，学生和教师的访问明显减少**，进一步提升了训练效率。

## 第七页：未来规划 + 致谢（1 分钟）

### 📖 演讲词：

最后，让我们展望一下**未来方向**。

我们规划了 **7 大核心方向**：

1. **概率计算独立化**——Old/Ref log prob 计算从推理引擎剥离，精确策略评估
2. **KV Cache 感知负载均衡**——根据 GPU 显存占用实时分配请求，消除热点与长尾
3. **低精度算力栈**——训练与推理全链路量化（MXFP8/NVFP4），显存与通信双降
4. **权重传输优化**——优化通讯带宽，参数同步时卸载 nccl/hccl 显存
5. **故障自愈与容错**——节点故障时自动迁移、恢复，整体训练不中断
6. **多轨迹单次 Rollout**——Agent Loop 一次 rollout 输出多条轨迹，提升样本效率
7. **推测解码协同训练**——Draft-Target model 协同推理 + 权重 refitting 联合优化

这些方向涵盖了从软件架构解耦到硬件感知调度，从集群弹性伸缩到算子级细粒度流水线的全栈优化。

## 附录：演讲技巧提示

| 技巧       | 说明                                                    |
|----------|-------------------------------------------------------|
| **节奏控制** | 第一页快（建立印象），第二页慢（讲清痛点），第三页适中（技术细节），第四~六页按内容密度调整        |
| **重点强调** | "2倍以上加速"、"准确率不降反升"、"4.3x加速"、"零拷贝" 等关键词需加重语气           |
| **互动时机** | 第二页长尾热力图处可停顿提问："大家猜猜看，长尾样本占多少比例？"                     |
| **手势指引** | 讲架构图时用手势指引数据流向；讲性能表格时用手指具体数字                          |
| **时间弹性** | 如时间充裕，第三页 Pipeline 图和第五页 TQ 架构图可展开详解；如时间紧张，可跳过第四页部分细节 |
| **邀约氛围** | 讲到生态邀约和昇腾邀约时放慢语速、提高音量，传递热情和开放的态度                          |
