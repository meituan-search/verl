## 第一页：VeRL 社区全异步历程（1 分钟）

VeRL 是目前主流的大模型强化学习训练框架，在当前获得了21k的star, 得到广大研究者的认可。
过去一年里，我们从传统的同步训练模式出发，逐步构建了全异步训练的生态体系，并且取得了显著的性能收益。
接下来，我会从异步训练的背景，构建，重构，生态以及规划这几个方面，向大家进行介绍。

## 第二页：（1.5 分钟）

为什么需要异步的强化学习？

已有的强化训练算法例如PPO、GRPO 或者DAPO中，核心迭代逻辑一致——每轮迭代依赖最新模型生成样本，等所有样本完成后再进行模型训练，并且进行参数更新。
这种模式保障稳定性，但也存在的效率瓶颈——在rollout阶段必须等最长的样本轨迹生成完成才能进行训练。

而随着大模型效果的提升，模型的CoT越来越长，以及工具和多轮对话在RL训练的参与, 我们面对的长尾问题越来越严重，关键在于：这种瓶颈无法通过加硬件解决，同步训练难以实现有效
scaling up。

而对于模型厂商来说，模型的迭代频次是日益增加，同步训练的效率瓶颈已经直接制约了产品迭代节奏，异步不是可选项，而是必选项。

给大家看一个宏观数据：DAPO 算法训练 32B 模型，单步总时长 1700 秒，Rollout 占 1200 秒（70%）。训练后期最大长度 20k vs 平均
5k，长尾问题非常显著。

再看微观视角：Qwen-235B 的单步 Rollout 总时长 30 分钟，其中超过一半时间在处理长尾样本。 部分 GPU资源 处于空闲的状态。

## 第三页：基础功能构建（2.5 分钟）⭐ 重点页

针对前面提到的长尾难题，我们基于 VeRL 框架，先后落地了 **One Step Off Policy** 和 **Fully Async Policy** 两种异步训练方案。

One Step Off Policy 的核心思路就是打破传统"样本生成完后再训练"的串行依赖——训练最开始先提前生成一个 step 的样本，
当前步训练所用的样本是上一步生成的，打破了原有 on-policy 的设定。

为了实现这种并行逻辑，需要对整体资源做合理划分：一部分专门分配给训练，另一部分分配给生成。
当然，这种模式也有缺点：生成所用的参数和训练所用的参数会固定相差一步，灵活性较差。

因此我们进一步设计了 Fully Async  架构——也就是**"生产者-消费者"模式的全异步流水线架构**：把 Rollouter
作为样本生产者，Trainer 作为样本消费者，两者完全并行运转。同时叠加异步样本陈旧度控制和 Partial Rollout 等特性，提升了
off-policy 策略的灵活性，最终取得接近 **3 倍的效率提升**。

初次以外，在算法层面，当前fully async 支持使用rollout log prob 直接进行 PPO 的loss计算，
，在训推不一致问题较为严重的场景下，我们也提供了 decoupled PPO 对 loss 进行修正。

最左面是一些实验数据，无论是7B模型，30B A3B, 以及我们内部的一些235B的实验，都取得至少2倍以上的性能的收益，在某一些长尾严重的场景下，收益的空间会更加突出。

## 第四页：基础设施重构（1.5 分钟）

而在框架层面，面对同步与异步双模式并行的复杂需求，以及算法持续演进带来的使用变更，需要通过不断重构框架，提升灵活性，可扩展性。

具体来说，RL 训练框架的4个重要组成部分，包括训练引擎、推理引擎、参数同步引擎、消息传输都进行了相应的重构工作。

对于训练引擎，我们抽象出了 Model Engine，统一了 SFT 和 RL 的训练框架代码，并且使用 Tinker 风格的 API 作为对外的接口，屏蔽具体引擎资源和并行的分配逻辑。
这样 RL 训练逻辑和引擎执行逻辑完全解耦，两者可以独立迭代。

对于推理引擎，统一使用 分进程的 Server-Client 架构，同时支持了 Partial Rollout auto abort/resume逻辑, AgentLoop 也就是工具执行过程无需感应中断，
同时将 AgentLoop 的管理 与 LLM Server 管理从代码层面进行解耦，提升框架的灵活性。

对于参数同步：抽象出 CheckpointEngine 作为一个可插拔的权重同步抽象层，提供统一的 API 来协调训练侧和推理侧之间的参数传输，屏蔽底层通信细节。
而实际的通讯工作，作为可扩展的后端支持不同设备的集合通讯或者P2P或者混合的通讯方式，也支持融入当前开源参数同步方案。

在消息传输方面，通过接入TransferQueue，实现了数据面/控制面分离、提高了整体集群的带宽利用率，降低了端到端训练耗时。

## 第五页：TransferQueue — 异腾核心贡献（1.5 分钟）

在这里，也要特别感谢**华为韩振宇同学**——他实现了 TransferQueue 这一高性能数据传输层，并成功接入了 VeRL 框架，提供了关键的数据总线基础设施。

在 VeRL 的 Single Controller 架构下，所有数据对象必须经过主控进行路由，这导致整个后训练系统存在一个单点瓶颈。

TransferQueue 正是为了解决这个问题而设计的。 它本质上是一个具备全景数据可见性和流式调度能力的高性能数据存储与传输模块。

从架构上看，TransferQueue 采用**控制面与数据面分离的双平面设计**：

**控制面（Control Plane）**由 TransferQueueController 负责，核心做几件事：
追踪每个训练样本的生产状态和消费状态、记录每个计算任务的消费历史、支持可定制的 Sampler 来灵活控制数据取用逻辑。不同计算任务可以独立消费同一份数据字段，互不干扰。

**数据面（Data Plane）**采用可插拔的存储后端设计。
默认使用 SimpleStorage 做 CPU 内存存储，开箱即用；
也支持昇腾原生的 Yuanrong 元荣系统做 HBM 到 SSD 的分层存储；
还支持 MooncakeStore 的 KV 分层存储和 Ray 的 Direct Transfer。用户可以根据自己的硬件环境灵活选择。

它具备四大核心特性：
第一是**零拷贝传输**，避免了数据序列化的开销；
第二是**方便扩展**，支持 TCP、RDMA 等多种传输协议；
第三是**细粒度访问**，可以做到子样本级别的读写操作和负载均衡；
第四是**生命周期管理**，支持主动式的资源回收和全景数据可见性。

最左面是一些实验数据也证明在真实场景下TQ的显著效率收益以及可靠性。

在 128 张 H100 的多模态后训练中，端到端性能提升达到了 **49.1%**，显著优化了 host 内存利用率和数据传输效率。
压力测试方面，8192 个并发客户端写入 2 TB 数据跨 4 节点，系统保持稳定无崩溃无丢失。

## 第六页：生态扩建（1 分钟）

### 📖 演讲词：

最后看生态扩建，主要包含两方面：一方面是软硬件多维度的适配，另一方面是开源功能的生态建设。

在底层支持上，VeRL 框架的异步训练逻辑原生支持 NVIDIA、Ascend 以及 AMD 硬件，并且正在积极适配更多国产芯片。
而在推理侧，异步训练也从最初只支持 vLLM，扩展覆盖到了 SGLang 和 TRT-LLM。
同时，训练框架也从 FSDP 覆盖到了 Megatron 和 MindSpeed。

在功能层面，这里受限于篇幅，向大家介绍三个核心的方向：
第一是 Agent Gateway——把推理服务抽象为 OpenAI 兼容的 API 接口，像 NeMo-Gym、Bedrock、SWE-agent 这些外部工具可以零代码接入。
第二是 Elastic Scheduling —— 实时监控队列深度，自动决策 GPU 资源在训练和推理之间动态切换，进一步减少 GPU 资源闲置率。
第三是 Online Policy Distillation——将OPD的逻辑与异步训练进行结合，减少训练中产生的bubble。
